## Общее описание

Messenger Application — это полнофункциональное приложение для обмена сообщениями, построенное на микросервисной архитектуре. Приложение позволяет пользователям регистрироваться, создавать приватные и публичные чаты, обмениваться сообщениями в реальном времени через WebSocket.

### Ключевые характеристики

- **Микросервисная архитектура**: каждый сервис отвечает за свою область ответственности
- **Асинхронная обработка сообщений**: через Apache Kafka
- **Real-time коммуникация**: WebSocket для мгновенной доставки сообщений
- **JWT аутентификация**: безопасный вход и авторизация
- **Кэширование**: Redis для хранения сессий и данных WebSocket

## Компоненты системы

### 1. Frontend (Порт 3000)

**Технологии**: React + TypeScript + Vite + Nginx

Веб-интерфейс приложения. Собранный React-проект раздается через Nginx, который также выступает как reverse proxy для API запросов.

**Основные функции**:
- Страница регистрации
- Страница входа
- Список чатов
- Чат с сообщениями
- Профиль пользователя
- WebSocket соединение для real-time сообщений

**Nginx конфигурация**:
```nginx
# Проксирование API запросов на gateway
location /api/ {
    proxy_pass http://gateway:8080/;
}

# Проксирование WebSocket
location /ws {
    proxy_pass http://gateway:8080/ws;
    proxy_set_header Upgrade $http_upgrade;
    proxy_set_header Connection "upgrade";
}
```

### 2. Gateway (Порт 8080)

**Технологии**: Go + Gin

Центральная точка входа для всех API запросов. Gateway маршрутизирует запросы к соответствующим микросервисам.

**Функции**:
- Маршрутизация запросов
- Аутентификация JWT токенов
- Проксирование к внутренним сервисам
- WebSocket upgrade и управление соединениями
- Kafka producer для отправки сообщений
- Kafka consumer для получения сообщений
- Интеграция с Redis для WebSocket данных

**Структура роутов**:
```
/auth/register    -> Auth Service
/auth/login       -> Auth Service
/user/me          -> User Service
/user/users       -> User Service
/user/id          -> User Service
/user/name        -> User Service
/chat/            -> Chat Service
/chat/private     -> Chat Service
/chat/public      -> Chat Service
/chat/:id/users   -> Chat Service
/ws               -> WebSocket (обрабатывается напрямую)
/message/upd      -> Gateway (заглушка)
/message/:id      -> Gateway (заглушка)
```

### 3. Auth Service (Порт 8081)

**Технологии**: Go + Gin + PostgreSQL

Сервис отвечает за аутентификацию и регистрацию пользователей.

**Функции**:
- Регистрация новых пользователей с хэшированием пароля
- Вход пользователя и генерация JWT токена
- Валидация данных пользователя

**Эндпоинты**:
- `POST /auth/register` — регистрация
- `POST /auth/login` — вход

**Процесс регистрации**:
1. Принимает username, email, password
2. Хэширует пароль с помощью bcrypt
3. Сохраняет пользователя в PostgreSQL
4. Возвращает ID созданного пользователя

**Процесс входа**:
1. Принимает username и password
2. Ищет пользователя в БД
3. Проверяет пароль
4. Генерирует JWT токен с помощью RSA ключей
5. Возвращает токен

### 4. User Service (Порт 8082)

**Технологии**: Go + Gin + PostgreSQL

Сервис управления пользователями.

**Функции**:
- Получение данных текущего пользователя
- Обновление профиля пользователя
- Получение списка всех пользователей
- Получение пользователя по ID
- Получение пользователя по username

**Эндпоинты**:
- `GET /me` — получить текущего пользователя (использует header `id`)
- `PUT /me` — обновить профиль
- `GET /users` — получить всех пользователей
- `POST /user/id` — получить пользователя по ID (JSON body)
- `POST /user/name` — получить пользователя по username (JSON body)

### 5. Chat Service (Порт 8083)

**Технологии**: Go + Gin + PostgreSQL

Сервис управления чатами.

**Функции**:
- Создание приватных чатов
- Создание публичных чатов
- Получение списка чатов пользователя
- Получение участников чата

**Эндпоинты**:
- `GET /chat/` — получить чаты пользователя
- `POST /chat/private` — создать приватный чат
- `POST /chat/public` — создать публичный чат
- `GET /chat/:chat_id/users` — получить участников чата

**Приватные чаты**:
- Создаются между двумя пользователями
- Имеют `is_private = true`
- Автоматически добавляются оба пользователя как участники

**Публичные чаты**:
- Создаются с названием и списком участников
- Имеют `is_private = false`
- Могут содержать множество участников

### 6. Message Service (Порт 8084)

**Технологии**: Go + Kafka + PostgreSQL

Сервис обработки сообщений. Работает полностью асинхронно через Kafka.

**Функции**:
- Получение сообщений из Kafka топика `message.send`
- Сохранение сообщений в БД
- Определение получателей сообщения
- Отправка уведомлений в топик `message.delivered`

**Kafka топики**:
- `message.send` — входящие сообщения от Gateway
- `message.delivered` — исходящие уведомления для Gateway

### 7. PostgreSQL (Порт 5432)

Реляционная база данных для хранения всех данных приложения.

**Таблицы**:

```sql
-- Пользователи
users (
    id UUID PRIMARY KEY,
    username VARCHAR(255) UNIQUE,
    email VARCHAR(255) UNIQUE,
    password_hash TEXT,
    created_at TIMESTAMPTZ,
    updated_at TIMESTAMPTZ
)

-- Чаты
chats (
    id UUID PRIMARY KEY,
    name VARCHAR(255),
    is_private BOOLEAN,
    created_at TIMESTAMPTZ,
    updated_at TIMESTAMPTZ
)

-- Участники чатов
chat_participants (
    chat_id UUID REFERENCES chats(id),
    user_id UUID REFERENCES users(id),
    joined_at TIMESTAMPTZ,
    PRIMARY KEY (chat_id, user_id)
)

-- Сообщения
messages (
    id UUID PRIMARY KEY,
    chat_id UUID REFERENCES chats(id),
    sender_id UUID REFERENCES users(id),
    content TEXT,
    is_delivered BOOLEAN,
    sent_at TIMESTAMPTZ
)
```

### 8. Redis (Порт 6379)

In-memory база данных для кэширования и хранения состояния WebSocket соединений.

**Использование**:
- Хранение маппинга `socketID -> userID` для WebSocket
- Проверка активности соединений
- Кэширование сессий

### 9. Apache Kafka (Порт 9092)

 распределённая платформа потоковой обработки сообщений.

**Топики**:
- `message.send` — отправка новых сообщений
- `message.delivered` — доставка сообщений получателям

### 10. Zookeeper (Порт 2181)

Служба координации для Kafka.

## Потоки данных

### 1. Регистрация пользователя

```
User -> Frontend -> Nginx -> Gateway -> Auth Service -> PostgreSQL
                                                              │
                                                              ▼
                                        Frontend <- Nginx <- Gateway <- Response
```

**Детали**:
1. Пользователь заполняет форму регистрации (username, email, password)
2. Frontend отправляет POST запрос на `/api/auth/register`
3. Nginx проксирует запрос на Gateway (8080)
4. Gateway проксирует запрос на Auth Service (8081)
5. Auth Service хэширует пароль и сохраняет в PostgreSQL
6. Ответ возвращается обратно по цепочке

### 2. Вход в систему

```
User -> Frontend -> Nginx -> Gateway -> Auth Service -> PostgreSQL
                                                              │
                                                              ▼
                                        Frontend <- Nginx <- Gateway <- JWT Token
```

**Детали**:
1. Пользователь вводит username и password
2. Frontend отправляет POST на `/api/auth/login`
3. Auth Service проверяет credentials в PostgreSQL
4. Auth Service генерирует JWT токен с помощью приватного RSA ключа
5. Токен сохраняется в localStorage браузера
6. Все последующие запросы включают токен в заголовке Authorization

### 3. Создание чата

```
User -> Frontend -> Nginx -> Gateway -> Chat Service -> PostgreSQL
                                                              │
                                                              ▼
                                        Frontend <- Nginx <- Gateway <- Chat ID
```

**Детали**:
1. Пользователь выбирает друга для приватного чата или создает публичный чат
2. Frontend отправляет POST на `/api/chat/private` или `/api/chat/public`
3. Chat Service создает запись в таблице `chats`
4. Chat Service добавляет участников в таблицу `chat_participants`
5. Возвращается ID созданного чата

### 4. Отправка сообщения (Real-time)

```
User -> WebSocket -> Gateway -> Kafka (message.send) -> Message Service
                                                              │
                                                              ▼
                                                        PostgreSQL
                                                              │
                                                              ▼
                                    WebSocket <- Gateway <- Kafka (message.delivered)
```

**Детали**:
1. Пользователь подключается к WebSocket на `/ws` с JWT токеном
2. Gateway проверяет токен и устанавливает соединение
3. Пользователь отправляет сообщение через WebSocket
4. Gateway отправляет сообщение в Kafka топик `message.send`
5. Message Service читает сообщение из Kafka
6. Message Service сохраняет сообщение в PostgreSQL
7. Message Service определяет получателей сообщения
8. Message Service отправляет уведомление в Kafka топик `message.delivered`
9. Gateway читает уведомление и отправляет сообщение получателю через WebSocket

## Аутентификация и авторизация

### JWT Токен

Приложение использует JWT токены для аутентификации. Токен содержит:
- `user_id` — ID пользователя
- `exp` — время истечения (24 часа)

Токен подписывается с использованием RSA-256 (асимметричное шифрование).

### Генерация ключей

```bash
# Генерация приватного ключа
openssl genrsa -out private.pem 2048

# Конвертация в формат PKCS8
openssl pkcs8 -topk8 -inform PEM -in private.pem -out private_pkcs8.pem -nocrypt

# Генерация публичного ключа
openssl rsa -in private.pem -pubout -out public.pem
```

### Middleware проверки токена

1. Извлекает токен из заголовка `Authorization: Bearer <token>` или query параметра `?token=<token>`
2. Парсит JWT токен с помощью публичного ключа
3. Извлекает `user_id` из claims
4. Сохраняет `user_id` в контексте запроса
5. Разрешает или блокирует доступ к защищенным эндпоинтам

## Работа с пользователями

### Получение данных пользователя

1. Запрос на `/user/me` с JWT токеном
2. Gateway извлекает `user_id` из токена
3. Gateway проксирует запрос на User Service с заголовком `id: <user_id>`
4. User Service ищет пользователя в PostgreSQL по ID
5. Возвращает данные пользователя

### Обновление профиля

1. PUT запрос на `/user/me` с данными для обновления
2. User Service обновляет запись в PostgreSQL
3. Если передан новый пароль — хэширует его

### Получение списка пользователей

1. GET запрос на `/user/users`
2. User Service возвращает всех пользователей из PostgreSQL
3. Пароли не возвращаются в ответе

---

## Работа с чатами

### Приватный чат

1. Пользователь A выбирает пользователя B для чата
2. Отправляется POST на `/chat/private` с `friend_id: B`
3. Chat Service проверяет, не существует ли уже приватный чат между ними
4. Создает новую запись в `chats` с `is_private = true`
5. Добавляет обоих пользователей в `chat_participants`

### Публичный чат

1. Пользователь отправляет POST на `/chat/public` с:
   - `name` — название чата
   - `participant_id` — массив ID участников
2. Chat Service создает запись в `chats` с `is_private = false`
3. Добавляет указанных участников в `chat_participants`

### Получение чатов пользователя

1. GET запрос на `/chat/`
2. Chat Service ищет все чаты, где пользователь является участником
3. Возвращает массив ID чатов

## Система сообщений и Kafka

### Архитектура сообщений

```
┌──────────────┐     message.send      ┌─────────────────┐
│   Gateway    │ ─────────────────────▶│  Message Service│
│  (Producer)  │                       │   (Consumer)    │
└──────────────┘                       └────────┬────────┘
                                                │
                                    ┌───────────┴───────────┐
                                    │                       │
                                    ▼                       ▼
                             PostgreSQL          message.delivered
                                                        │
                                                        ▼
                                                ┌──────────────┐
                                                │   Gateway    │
                                                │  (Consumer)  │
                                                └──────┬───────┘
                                                       │
                                                       ▼
                                              WebSocket получателю
```

### Обработка сообщений

1. **Отправка**:
   - Пользователь отправляет сообщение через WebSocket
   - Сообщение содержит: `chat_id`, `sender_id`, `content`, `sent_at`
   - Gateway отправляет в Kafka топик `message.send`

2. **Обработка**:
   - Message Service читает из `message.send`
   - Сохраняет в PostgreSQL таблицу `messages`
   - Определяет всех участников чата (кроме отправителя)
   - Для каждого участника создает уведомление

3. **Доставка**:
   - Message Service отправляет уведомление в топик `message.delivered`
   - Gateway читает уведомление
   - Находит WebSocket соединение получателя
   - Отправляет сообщение через WebSocket

## WebSocket соединения

### Установка соединения

1. Клиент подключается к `ws://host:8080/ws?token=<JWT>`
2. Gateway проверяет JWT токен
3. Извлекает `user_id` из токена
4. Создает уникальный `socketID` (UUID)
5. Сохраняет маппинг в Redis:
   - `socket:<user_id>` -> `socketID`
   - `socket_to_user:<socketID>` -> `userID`
6. Обновляет HTTP соединение до WebSocket

### Обмен сообщениями

1. Клиент отправляет сообщение в формате JSON:
   ```json
   {
     "type": "send_message",
     "data": {
       "chat_id": "uuid-чата",
       "sender_id": "uuid-отправителя",
       "content": "текст сообщения",
       "sent_at": 1234567890
     }
   }
   ```

2. Gateway проверяет активность соединения через Redis
3. Пересылает сообщение в Kafka

### Получение сообщений

1. Gateway читает из Kafka топика `message.delivered`
2. Парсит `MessageDelivery` структуру
3. Ищет WebSocket соединение получателя по `userID` из Redis
4. Отправляет сообщение через WebSocket

### Закрытие соединения

1. Клиент закрывает соединение или происходит ошибка
2. Gateway удаляет маппинги из Redis
3. Удаляет соединение из памяти

## База данных

### Схема данных

```
┌─────────────┐       ┌─────────────┐       ┌─────────────┐
│    users    │       │    chats    │       │  messages   │
├─────────────┤       ├─────────────┤       ├─────────────┤
│ id (PK)     │       │ id (PK)     │       │ id (PK)     │
│ username    │       │ name        │       │ chat_id (FK)│
│ email       │       │ is_private  │       │ sender_id(FK)│
│ password_hash       │ created_at  │       │ content     │
│ created_at  │       │ updated_at  │       │ is_delivered│
│ updated_at  │       └──────┬──────┘       │ sent_at     │
└─────────────┘              │              └─────────────┘
        │                    │
        │                    │
        │       ┌────────────┴────────────┐
        │       │  chat_participants      │
        │       ├─────────────────────────┤
        └──────▶│ chat_id (FK, PK)        │
                │ user_id (FK, PK)        │
                │ joined_at               │
                └─────────────────────────┘
```

### Связи

- `chats` -> `chat_participants` (one-to-many)
- `users` -> `chat_participants` (one-to-many)
- `chats` -> `messages` (one-to-many)
- `users` -> `messages` (one-to-many)

## Кэширование

### Redis

Используется для хранения состояния WebSocket соединений:

1. **Проверка активности**: перед обработкой каждого сообщения Gateway проверяет, существует ли ключ `socket:<user_id>` в Redis. Если ключ отсутствует — соединение считается закрытым.

2. **Маппинг соединений**: позволяет быстро находить WebSocket соединение конкретного пользователя для доставки сообщений.

3. **Время жизни**: ключи живут 30 минут (timeout соединения).

## Безопасность

1. **Аутентификация**: JWT токены с RSA шифрованием
2. **Пароли**: Хэширование с bcrypt
3. **WebSocket**: Проверка токена при каждом соединении
4. **API**: Все защищенные endpoints требуют валидный JWT
5. **База данных**: Приватные ключи хранятся в файловой системе контейнера
